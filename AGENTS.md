本文件是 599549.com 项目的仓库级开发规则。
Codex / AI 工程代理在本仓库内执行任何任务前，必须先读取本文件。

项目路径建议：

/www/wwwroot/599549.com/AGENTS.md


# 一、规则优先级

执行优先级：

1. 用户当前明确任务
2. 本仓库 AGENTS.md
3. 常驻自定义指令
4. Codex 默认行为

如果 AGENTS.md 与常驻自定义指令冲突：

以 AGENTS.md 为项目执行准则。

如果用户当前任务与 AGENTS.md 冲突：

用户当前任务优先，但不得违反以下红线：

1. 数据安全
2. 澳门 / 香港数据隔离
3. 登录权限保护
4. 开奖逻辑保护
5. 禁止危险命令

如果“模块闭环”和“最小修改”冲突：

优先完成当前任务必要闭环，但不得跨无关模块。

如果“继续推进”和“信息不足”冲突：

能从现有代码判断的，按现有代码推进。
不能从现有代码判断的，不得自由设计，必须说明缺失信息。

如果“UI 修复”和“稳定区域保护”冲突：

只允许修复当前问题，不允许重做稳定区域结构。

# 五、任务类型

一、只读排查

触发关键词：

排查、分析、检查、为什么、找原因、看日志、不要修改。

规则：

1. 只允许查看、搜索、分析。
2. 禁止修改任何文件。
3. 输出原因、证据、涉及文件、建议方案。


二、低风险修复

触发关键词：

修复、BUG、报错、异常、失效、不显示、错位、溢出、卡顿。

规则：

1. 只修改问题直接相关文件。
2. 不扩大范围。
3. 不顺手重构。
4. 不顺手优化无关模块。
5. 修复后必须验证对应页面或功能。


三、模块开发

触发关键词：

开发、新增、完成、补全、继续做、完善业务。

规则：

1. 按一个完整模块闭环推进。
2. 可以一次修改多个相关文件。
3. 相关文件必须属于同一个业务模块。
4. 不做碎片式修改。
5. 不跨无关模块。


四、UI 修复

触发条件：

用户发截图、指出视觉问题、说布局错乱、比例不协调、手机端异常。

规则：

1. 必须先定位对应代码。
2. 只修指定区域及必要相关区域。
3. 不得凭截图扩展到无关模块。
4. 不得重做设计。
5. 不得改变整体风格。


# 六、禁止碎片式开发

禁止以下方式：

1. 只改一个按钮就结束。
2. 只改一行 CSS 就结束。
3. 只修一个角落不看当前模块整体。
4. 只做澳门不看香港。
5. 只做香港不看澳门。
6. 只做前台不看后台。
7. 只写页面不接数据。
8. 只接数据不处理空状态。
9. 只做 PC 不看手机端。
10. 做一步停下来问一步。

正确方式：

1. 一个页面、一个功能、一个后台模块、一个 UI 区块，必须形成当前任务范围内的完整闭环。
2. 当前需求直接相关的前台、后台、API、数据处理、样式、空状态、错误状态、移动端适配，应一次处理完整。
3. 能从现有代码和同类页面判断的，直接按现有结构推进。
4. 信息不足时，不允许自由设计，只能基于现有代码和同类页面做最小一致性补全。
5. 只有高风险事项才停下来确认。


# 七、修改范围规则

允许一次修改多个文件，但必须属于同一个业务模块。

禁止：

1. 跨模块顺手修改。
2. 顺手重构。
3. 顺手优化。
4. 批量格式化全项目。
5. 批量替换未知范围代码。
6. 删除未确认文件。
7. 清空日志。
8. 清空缓存。
9. 删除上传目录。
10. 改动未确认的稳定区域。

如果预计修改超过 3 个业务文件，必须先说明原因和必要性。


# 八、澳门 / 香港双前台规则

必须始终同时考虑澳门和香港。

规则：

1. 澳门有的功能，必须检查香港是否需要对应。
2. 香港有的功能，必须检查澳门是否需要对应。
3. 共用后台必须同时服务澳门和香港。
4. 数据必须使用项目现有 region / site / system 等区域字段区分。
5. 不得让澳门数据覆盖香港数据。
6. 不得让香港数据覆盖澳门数据。
7. 不得只修澳门漏掉香港。
8. 不得只修香港漏掉澳门。
9. 不得无理由制造两套不同代码。
10. 前台可以有地区文案差异，但结构和功能逻辑必须保持一致。

允许不同：

1. 系统名称
2. 品牌标识
3. 地区文案
4. 开奖时间
5. 开奖期号
6. 开奖结果
7. 开奖数据来源
8. 开奖历史记录

3. 共用澳门 / 香港开奖期号。
4. 因后台共用导致开奖数据串区。
5. 不涉及开奖时误改开奖逻辑。

凡是涉及开奖，必须明确说明：

1. 是否影响澳门开奖。
2. 是否影响香港开奖。
3. 是否影响开奖时间。
4. 是否影响开奖期号。
5. 是否影响开奖结果。
6. 是否影响开奖数据来源。
7. 是否影响历史记录。

# 十二、UI / 布局 / 视觉规则

UI 修改必须以以下来源为准：

1. 现有代码结构
2. 用户截图明确指出的问题
3. 同类页面已有样式
4. 项目现有 CSS 规则

禁止：

1. 凭感觉重新设计。
2. 改变页面整体结构。
3. 改变模块顺序。
4. 改变信息层级。
5. 改卡片结构。
6. 改字体体系。
7. 改整体风格。
8. 大量使用 !important。
9. 随意修改全局 CSS。
10. 澳门一套 UI，香港一套 UI。
11. 为了美观破坏功能。
12. 为了塞内容把字体改得过小。
13. 用固定高度硬压动态内容。
14. 图片拉伸变形。

允许：

1. 修错位。
2. 修溢出。
3. 修间距。
4. 修对齐。
5. 修图片变形。
6. 修手机端显示异常。
7. 在同类页面已有样式基础上补齐缺失样式。

UI 修复不能只改一个点，必须检查当前模块整体协调：

1. 顶部是否正常。
2. 内容区是否正常。
3. 卡片比例是否正常。
4. 字体层级是否正常。
5. 按钮大小是否正常。
6. 图片是否变形。
7. 列表是否错位。
8. 底部导航是否遮挡内容。
9. 手机端是否横向溢出。
10. 澳门 / 香港对应页面是否一致。


# 十三、移动端标准

移动端优先。

必须满足：

1. 页面不得横向滚动。
2. 主体内容不得贴边。
3. 底部导航不得遮挡内容。
4. 点击区域不得过小。
5. 文字不得挤成一团。
6. 卡片不得被压扁。
7. 长标题必须换行或省略，不能撑爆布局。
8. 图片不得拉伸变形。

常用尺寸参考：

1. 页面左右边距：10px - 14px。
2. 卡片内边距：8px - 12px。
3. 模块上下间距：8px - 16px。
4. 正文字号：13px - 15px。
5. 辅助文字：12px - 13px。
6. 标题字号：16px - 20px。
7. 重要数字：18px - 26px。
8. 按钮高度：36px - 44px。
9. 底部导航点击区不低于 44px。


# 十四、CSS / HTML 治理规则

CSS 修改必须遵守优先级：

1. 组件 CSS
2. 模块 CSS
3. 页面 CSS
4. 全局 CSS

禁止优先修改全局 CSS，除非问题确实来自全局样式。

禁止：

1. 大量新增 !important。
2. 重复创建功能相同的 class。
3. 多个文件反复覆盖同一个选择器。
4. 用临时 CSS 掩盖 HTML 结构问题。
5. 全站搜索替换未知影响的 class。
6. 为一个页面问题改全局布局。
7. 批量重排 CSS 文件。
8. 批量格式化 CSS 文件。

HTML 修改必须遵守：

1. 不新增无意义 div 嵌套。
2. 不复制大量重复结构。
3. 不改变原有模块顺序。
4. 不改变信息层级。
5. 不破坏澳门 / 香港结构一致性。
6. 不用 CSS 掩盖 HTML 结构错误。

发现 CSS 或 HTML 重复时，不能直接大清理，必须先说明：

重复位置：
影响范围：
建议保留方案：
删除风险：
是否本次必须处理：

# 十七、数据库和接口规则

默认禁止：

1. 修改数据库结构。
2. 修改字段名称。
3. 修改字段含义。
4. 修改接口地址。
5. 修改请求方式。
6. 修改请求参数。
7. 修改返回结构。
8. 修改权限逻辑。
9. 修改登录逻辑。
10. 修改开奖数据逻辑。

允许数据库只读审计。

涉及业务开发、后台开发、数据不显示、澳门 / 香港串区、列表异常、筛选异常时，允许先做数据库只读审计。

只读审计允许查看：

1. 表名。
2. 字段。
3. 索引。
4. region / site / system 区域字段。
5. 少量样本数据。
6. 数据关联关系。

只读审计禁止：

1. 修改表结构。
2. 删除数据。
3. 更新数据。
4. 清空数据。
5. 执行不确定 SQL。
6. 修改字段含义。

如果必须修改数据库或接口，必须先输出：

1. 为什么必须改。
2. 当前结构为什么不够。
3. 准备改什么。
4. 影响澳门什么。
5. 影响香港什么。
6. 影响后台什么。
7. 回滚方式。
8. SQL 或接口变更草案。

# 二十、性能规则

用户说慢、卡、加载久、反应迟钝时，不能直接改 UI。

必须优先检查：

1. 404 是否刷屏。
2. 500 是否存在。
3. PHP Fatal 是否存在。
4. 日志是否疯狂增长。
5. API 是否重复请求。
6. JS 轮询是否过快。
7. 数据库是否慢查询。
8. 页面资源是否过大。
9. CSS / JS 是否重复加载。
10. 后台列表是否没有分页。
11. 缓存文件是否异常。
12. 服务器 CPU / 内存 / 磁盘 / inode 是否异常。

没有证据，不允许盲目优化。

普通功能开发时，禁止顺手做性能优化。

禁止顺手修改：

1. 缓存策略。
2. SQL 结构。
3. 索引。
4. JS 加载机制。
5. 资源合并方式。
6. 轮询频率。
7. 全局加载流程。
8. 后台分页逻辑。

只有当本次任务明确是性能问题，或已发现证据时，才允许处理性能。


# 二十一、Git 规则

修改前必须执行：

git status --short

如果工作区不干净：

1. 不要覆盖无关改动。
2. 判断哪些文件与本次任务相关。
3. 只修改本次模块相关文件。
4. 如果会覆盖用户已有改动，必须停止说明。

禁止执行：

1. git reset
2. git clean
3. rm -rf
4. 删除上传目录
5. 清空数据库
6. 清空日志
7. 清空缓存
8. 重启服务
9. 修改 stable 标签

除非用户明确授权。

提交规则：

1. 未经用户要求，不自动提交。
2. 没有真实提交，不得输出提交号。
3. 已提交才输出真实 commit hash。
4. 未提交必须写：未提交。


# 二十二、禁止批量格式化规则

除非用户明确要求，禁止执行：

1. 全项目格式化。
2. 全 CSS 格式化。
3. 全 PHP 格式化。
4. 全 JS 格式化。
5. 大规模自动重排代码。
6. 批量替换未知范围代码。
7. 删除未确认的旧代码。
8. 清理未确认的历史文件。

修改文件时，只允许改本次任务直接相关代码段。

如果工具自动格式化导致大量 diff，必须停止并说明，不得继续提交。


# 二十三、修改回滚保护规则

每次修改必须保证可追踪、可恢复、可回滚。

修改前必须确认：

1. Git 状态。
2. 当前分支。
3. 相关文件是否已有用户改动。
4. 是否会覆盖已有改动。

修改时必须做到：

1. 不做不可逆操作。
2. 不删除业务文件。
3. 不清空目录。
4. 不覆盖用户未提交改动。
5. 不修改无关模块。
6. 不跨区域写入数据。

如果修改失败，必须说明：

1. 已改哪些文件。
2. 哪些文件可能需要回退。
3. 如何回滚。
4. 是否有数据影响。

禁止在未授权情况下执行：

git reset
git clean
rm -rf
数据库删除
清空缓存
清空日志
重启服务


# 二十四、完成标准 Definition of Done

每个任务完成时，必须满足当前范围内的完成标准。

前台页面完成标准：

1. 页面能打开。
2. 数据能正常显示。
3. 空数据状态正常。
4. 错误状态合理。
5. 手机端不横向溢出。
6. PC 端不明显错位。
7. 顶部正常。
8. 底部导航正常。
9. 澳门 / 香港对应页面结构一致。
10. 不影响开奖卡片和公告区，除非本次明确修改。

后台模块完成标准：

1. 列表能打开。
2. 分页正常。
3. 筛选正常。
4. 新增正常。
5. 编辑正常。
6. 删除或状态切换正常。
7. region 写入正确。
8. 澳门 / 香港数据不串区。
9. 权限未绕过。
10. 登录未受影响。

API 完成标准：

1. 请求方式正确。
2. 参数兼容。
3. 返回结构兼容。
4. 错误处理明确。
5. 不泄露路径、SQL、密钥、token。
6. 澳门 / 香港数据隔离正确。

UI 完成标准：

1. 当前模块比例协调。
2. 字体层级清楚。
3. 间距统一。
4. 图片不变形。
5. 按钮可点击。
6. 手机端不溢出。
7. 底部导航不遮挡。
8. 澳门 / 香港视觉结构一致。

未达到完成标准时，不得说“已完成”，只能说“已完成部分”和“未完成原因”。


# 二十五、验证规则

每次修改后必须按实际范围验证。

前台修改至少验证：

1. 澳门对应页面。
2. 香港对应页面。
3. 手机端显示。
4. PC 显示。
5. 顶部是否正常。
6. 底部导航是否正常。
7. 是否横向溢出。
8. 是否白屏。
9. 是否 404。
10. 是否 500。
11. 是否影响开奖卡片。
12. 是否影响公告区。

后台修改至少验证：

1. 后台页面能打开。
2. 列表能显示。
3. 分页能使用。
4. 筛选能使用。
5. 新增 / 编辑 / 删除 / 状态切换按本次范围验证。
6. 澳门 / 香港数据不串区。
7. 权限没有被破坏。
8. 登录没有被影响。

API 修改至少验证：

1. 请求是否成功。
2. 返回结构是否兼容。
3. 错误处理是否正常。
4. 是否影响澳门。
5. 是否影响香港。
6. 是否泄露敏感信息。

advanced_analysis 修改至少验证：

1. 每次调整 advanced_analysis 引擎、规则配置、历史读取或回测脚本后，必须执行固定验收命令：

php80 tools/advanced_analysis_boundary_backtest.php

2. 必须记录澳门 / 香港最小样本、临界样本、大样本的命中率、耗时、内存。
3. 如果当前环境无法连接真实数据库，不得说已通过，必须说明无法验证原因，并给出用户可复测命令。
4. 不得为了通过验收触发数据库写入、远程同步或 UI 接入。

无法验证时，不得假装验证成功，必须写清楚：

1. 已验证什么。
2. 未验证什么。
3. 为什么无法验证。
4. 建议用户如何复测。

# 二十七、二次开发工作方式

本项目是半成品二次开发，不是从零开发。

默认工作方式：

1. 先收敛现有混乱。
2. 再补齐缺失功能。
3. 再统一澳门 / 香港。
4. 再完善后台闭环。
5. 最后才做优化和美化。

禁止：

1. 推翻重做。
2. 新旧结构并存。
3. 每次新增一套写法。
4. 为了快制造临时代码。
5. 为了美观破坏业务。
6. 为了性能提前重构。

正确方式：

1. 找已有同类模块。
2. 复用已有结构。
3. 补齐当前模块缺口。
4. 保持澳门 / 香港一致。
5. 保持后台共用。
6. 保持数据隔离。
7. 完成后验证。


# 二十八、核心纪律

不重构，只演进。
不乱猜，只看代码。
不碎改，只做模块闭环。
不自由设计，只基于现有结构修复。
不只做澳门漏香港。
不只做香港漏澳门。
不只做前台漏后台。
不改登录权限。
不乱碰开奖。
不编造结果。
不假装验证。
不扩大范围。
不破坏 stable。
