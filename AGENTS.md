# 599549.com AGENTS.md（业务交付执行版）

本文件是 599549.com 的仓库级开发规则。Codex 或其他 AI 工程代理在本仓库执行任务前必须先读取本文件。

建议路径：

`/www/wwwroot/599549.com/AGENTS.md`

---

## 一、项目身份与目标

项目结构：

1. 澳门前台系统。
2. 香港前台系统。
3. 共用后台管理系统。

现有技术栈以仓库实际代码为准，默认包括：

1. PHP。
2. MySQL / MariaDB。
3. 原生 JavaScript。
4. HTML / CSS。
5. 现有前后台结构和接口方式。

本项目属于半成品二次开发维护，不是从零开发。

目标：

1. 基于当前 stable 稳定基线继续开发。
2. 补齐业务功能和业务链路。
3. 主动收敛现有混乱。
4. 保持澳门、香港数据隔离和开奖独立。
5. 推进到可验证、可部署、可维护、可上线状态。

禁止推翻重写、并行制造第二套系统或擅自迁移技术栈。

---

## 二、规则优先级

执行优先级：

1. 安全红线、数据安全和法律合规要求。
2. 用户当前明确业务目标。
3. 本 AGENTS.md。
4. 常驻自定义指令。
5. Codex 默认行为。

用户当前任务可以定义本次业务目标和范围，但不得覆盖：

1. 数据安全。
2. 澳门 / 香港数据隔离。
3. 登录与权限保护。
4. 开奖独立保护。
5. 危险命令限制。
6. 禁止非法业务功能。

如果规则冲突：

1. 先保证数据、安全、登录权限和开奖独立。
2. 再完成用户当前业务目标。
3. 不能安全判断时，仅停止高风险部分；其他可安全实施的部分继续推进。
4. 必须明确说明阻断点，不得把整个任务无条件停掉。

## 三、业务交付责任

用户只需描述：

1. 需要什么业务功能。
2. 使用场景。
3. 最终应达到什么效果。
4. 明确的限制或差异。

用户不负责告诉 Codex：

1. 应修改哪些文件。
2. 应使用哪个函数。
3. 应查询哪张表。
4. 前后台如何连接。
5. 页面、接口和数据库如何联动。
6. 应测试哪些关联入口。

这些属于工程实现责任，必须由 Codex通过现有代码主动识别和完成。

每个业务目标必须形成“业务交付包”。交付包可以跨多个相关文件和模块，不受固定文件数量限制，但只能包含实现当前业务目标直接需要的内容。

业务交付包按实际需要覆盖：

1. 澳门前台入口。
2. 香港前台入口。
3. 共用后台。
4. PHP 控制器、服务、模型或现有等效层。
5. API 或表单处理链路。
6. 数据读取、写入和区域隔离。
7. HTML、CSS、JavaScript。
8. 列表、详情、分页、筛选、搜索。
9. 新增、编辑、状态切换、回收站。
10. 加载、空数据、失败和权限状态。
11. PC 和移动端。
12. 安全校验。
13. 回归测试。

不是每个业务都必须机械包含全部项目；凡是当前业务直接关联的部分，必须一次处理完整。

首次进入项目、新会话、换目录、换分支或接手未知任务时，先执行或等效检查：

```bash

git status --short
git status -sb
git remote -v
git branch --show-current
git log -5 --oneline
```

必须识别：

4. 澳门前台入口。

工作区不干净时：

2. 不覆盖、不回退、不清理用户改动。

4. 不重叠时继续实施安全部分。
5. 会覆盖或产生无法区分的冲突时，才停止相关文件并说明。

## 五、默认执行流程

每个业务任务按以下流程执行：

1. 读取 AGENTS.md 和相关代码。
2. 确认当前业务目标。
3. 找到现有同类实现、入口和数据链路。
4. 建立影响矩阵：
   - 澳门前台
   - 香港前台
   - 共用后台
   - API / 表单
   - 数据库
   - 权限 / 登录
   - 开奖
   - PC / 移动端
5. 确定当前任务必须完成的业务交付包。
6. 实施所有直接相关修改。
7. 检查遗漏的关联入口、空状态、错误状态和区域隔离。
8. 执行验证和回归。
9. 修正本次改动引入的问题。
10. 输出实际完成、未验证项、风险和 Git 状态。

默认不要做一步问一步。

只有高风险事项需要先确认。普通页面、后台功能、PHP 修复、UI 修复、低风险批量修复和相关多文件联动应直接完成。

## 六、任务模式

### 1. 只读排查

用户明确说“只排查、只分析、不要修改”时：

1. 只读取和分析。
2. 不修改文件和数据。
3. 输出根因、证据、影响范围和建议处理顺序。

### 2. 业务开发

用户说“开发、新增、补全、完成、继续做、完善”时：

1. 直接进入业务交付模式。
2. 主动识别所有直接关联层。
3. 一次完成当前业务闭环。
4. 不等待用户逐项指出前台、后台、接口和移动端遗漏。

### 3. 低风险修复

用户说“修复、报错、异常、失效、不显示、错位、卡顿”时：

1. 找到根因。
2. 修复当前根因涉及的完整链路。
3. 不只修补表面症状。
4. 不顺手扩大到无关业务。

### 4. 低风险批量修复

完成审计后，存在同一根因、同一风险、同一修复方式和同一验证标准的问题时，可以跨多个相关文件或模块一次处理。

不得纳入批量修复：

1. 数据库结构和字段语义。
2. 接口契约。
3. 登录和权限核心。
4. 开奖核心。
5. 澳门 / 香港区域数据语义。
6. 用户数据和 uploads。

### 5. UI / 截图任务

用户提供截图或描述视觉问题时：

1. 先定位对应页面和真实代码。
2. 判断问题是 HTML、CSS、JavaScript、数据状态还是资源加载造成。
3. 处理当前区域及其必要关联布局。
4. 不凭截图重做整个页面。
5. 不只让截图区域暂时好看而破坏整体、PC、移动端或另一地区页面。

### 6. 上线、部署和安装包

涉及上线、打包、迁移或安装器时：

1. 先区分空站安装包和业务完整安装包。
2. 不得只创建程序骨架和空表后声称业务完整。
3. 必须按本文件“部署与业务完整性”章节执行。

## 七、主动关联检查

完成用户目标时，必须主动检查：

1. 入口是否已加入或可访问。
2. 澳门和香港是否都需要对应入口。
3. 共用后台是否需要列表、筛选、编辑或状态管理。
4. API 或表单请求是否完整。
5. region / site / system 是否正确透传和过滤。
6. 空数据、失败、无权限和未登录状态是否合理。
7. 分页、搜索、筛选是否与业务一致。
8. 前端状态与后台状态是否同步。
9. 手机端和 PC 是否都可用。
10. 公共组件或样式是否受影响。
11. 是否引入 404、500、PHP Fatal、JavaScript 报错或重复请求。
12. 是否需要部署配置、定时任务或初始化数据。

不得等用户发现以上遗漏后再补。

## 八、澳门 / 香港双前台规则

必须保持：

1. 共用后台统一管理。
2. 澳门与香港数据严格隔离。
3. 使用项目现有 `region`、`site`、`system` 或等效区域字段。
4. 澳门写入不得进入香港。
5. 香港写入不得进入澳门。
6. 对应功能结构和业务逻辑原则上一致。
7. 不得只做一个地区而遗漏另一个地区。

允许不同：

1. 系统名称。
2. 品牌标识。
3. 地区文案。
4. 开奖时间。
5. 开奖期号。
6. 开奖结果。
7. 开奖数据来源。
8. 开奖历史记录。

如果现有项目确实采用两套不同入口或模板，应沿用现状并保持行为一致，不得为了形式统一强行重构。

## 九、开奖独立规则

开奖属于高风险业务。

未经用户明确确认，不得修改：

1. 澳门开奖时间。
2. 香港开奖时间。
3. 澳门开奖期号。
4. 香港开奖期号。
5. 澳门开奖结果。
6. 香港开奖结果。
7. 数据来源。
8. 历史记录。
9. 同步和解析核心逻辑。

禁止：

1. 合并澳门和香港开奖记录。
2. 共用时间、期号或结果。
3. 因后台共用导致数据覆盖。
4. 无关任务顺手修改开奖代码。

涉及开奖时必须说明实际影响，并分别验证澳门和香港。

## 十一、UI、布局和视觉规则

UI 必须基于：

1. 当前 HTML。
2. 当前 CSS。
3. 当前 JavaScript。
4. 同类页面。
5. 用户截图和明确要求。
6. 项目已有组件和变量。

必须主动检查：

1. 页面整体比例。
2. 内容宽度。
3. 模块间距。
4. 卡片内边距。
5. 字体层级。
6. 按钮和点击区域。
7. 图片比例。
8. 长文本。
9. 列表和表格。
10. 顶部、底部导航和固定元素。
11. 横向溢出。
12. 澳门 / 香港对应页面。
13. PC 和移动端。

禁止：

1. 为了修一点而新增一层覆盖补丁。
2. 大量使用 `!important`。
3. 用固定高度压动态内容。
4. 随意修改全局 CSS。
5. 改变无关模块的视觉。
6. 图片拉伸。
7. 澳门和香港无理由使用不同结构。
8. 只在单一视口下验收。

常用尺寸只能作为参考，项目现有稳定体系优先。

后台字体变量继续使用项目现有约定：

1. 表头正文使用 `--admin-head-font-weight`。
2. 表单控件正文使用 `--admin-control-font-weight`。
3. 不为同类文本重复硬编码局部权重。

## 十二、HTML、CSS 和 JavaScript 治理

### HTML

1. 不新增无意义嵌套。
2. 不复制大量相同结构。
3. 不改变无关模块顺序和信息层级。
4. 不用 CSS 掩盖结构错误。
5. 保持现有 JavaScript 选择器和表单字段兼容。

### CSS

优先级：

1. 组件 CSS。
2. 模块 CSS。
3. 页面 CSS。
4. 全局 CSS。

规则：

1. 先查最终生效规则和冲突来源。
2. 修正根因，不在文件末尾无限追加覆盖。
3. 不批量格式化整个 CSS。
4. 删除重复或旧规则前必须确认引用和最终 computed style。
5. 共用组件修改后必须检查澳门、香港、后台及相关页面。

### JavaScript

1. 不重复绑定事件。
2. 不重复请求 API。
3. 不制造无限重试或过快轮询。
4. 清理当前任务引入的定时器和状态。
5. 错误必须有可控反馈。
6. 不改变无关页面全局行为。
7. 不保留调试输出和秘密信息。

后台是共用后台，必须服务澳门和香港。

1. 列表。
2. 分页。
3. 搜索和筛选。
4. 澳门 / 香港区域筛选。
5. 新增。
6. 编辑和回显。
7. 删除、回收站或状态切换。
8. 排序。
9. 表单验证。
10. 错误和成功反馈。
11. 权限检查。
12. 数据区域写入和查询。

禁止：

1. 重写后台架构。
2. 绕过现有权限。
3. 绕过现有数据库封装或接口方式。
4. 每个模块制造一套不同交互。
5. 未经确认修改登录和权限核心。

## 十五、安全规则

所有外部输入默认不可信。

必须防范：

1. SQL 注入。
2. XSS。
3. CSRF。
4. 越权和 IDOR。
5. 文件上传风险。
6. 上传目录执行 PHP。
7. 路径穿越和文件包含。
8. 登录暴力尝试。
9. Session 和 Cookie 风险。
10. 敏感信息泄露。
11. 澳门 / 香港数据越区访问。
12. 错误页面暴露路径、SQL、token 或密钥。

不得开发：

1. 下注。
2. 赔率。
3. 彩票销售。
4. 购彩。
5. 提现。
6. 返奖。
7. 资金结算。
8. 赌博交易。
9. 赌博资金流转。
10. 非法引流。

会员、积分和权限仅可用于站内内容和功能访问控制。

## 十六、性能规则

出现慢、卡、加载久、反应迟钝时，先收集证据：

1. 404、500、PHP Fatal。
2. 日志异常增长。
3. API 重复请求。
4. JavaScript 轮询。
5. 数据库慢查询和 N+1。
6. 无分页大列表。
7. CSS / JavaScript 重复加载。
8. 图片和静态资源大小。
9. CPU、内存、磁盘和 inode。
10. 缓存目录和权限。

没有证据不得盲目重构。

普通业务开发不顺手改变缓存、索引、加载架构或资源构建流程；当前业务确实受其阻断时，先说明证据和影响。

---

## 十七、低风险批量修复

允许按一个问题组批量处理：

1. 同类静态资源 404。
2. 同类 PHP Notice 或非业务 Warning。
3. 重复资源加载。
4. 重复事件绑定或重复请求。
5. 明确无效的调试输出。
6. 明确被覆盖且无作用的重复 CSS。
7. 明确无引用、非用户数据、可恢复的临时残留。
8. 同一根因造成的响应式或兼容性问题。

进入条件：

1. 同一根因。
2. 同一风险等级。
3. 同一修复方式。
4. 同一验证标准。
5. 不改变业务语义。
6. 不涉及数据库结构、接口契约、登录权限、开奖核心和区域数据语义。

每个问题组处理完成后立即回归，不能混入其他类型问题。

## 十八、Git 和文件保护

修改前必须执行：

```bash
git status --short
```

禁止未经授权执行：

1. `git reset`。
2. `git clean`。
3. `rm -rf`。
4. 删除 uploads。
5. 清空数据库。
6. 清空日志、缓存或 Session。
7. 重启服务。
8. 修改 stable 标签。
9. 全项目格式化。
10. 批量替换未知范围代码。

提交规则：

3. 用户要求提交时，只提交当前业务交付包的相关文件。

5. 未提交写“未提交”。

## 十九、验证和批量回归

验证按实际影响矩阵执行，不做无意义全站假测试。

最低冒烟项：

1. 澳门对应入口。
2. 香港对应入口。
3. 共用后台相关入口。
4. API 或表单链路。
5. 无新增 404、500、PHP Fatal。
6. 无新增 JavaScript 错误。
7. 澳门 / 香港不串区。
8. 登录、权限和开奖未受意外影响。
9. Git diff 与任务范围一致。

前台业务按需验证：

1. 页面可访问。
2. 数据、空状态和错误状态。
3. PC 和移动端。
4. 顶部、底部和固定元素。
5. 360、390、768、1366 等关键视口。

后台业务按需验证：

1. 列表、分页和筛选。
2. 新增、编辑和状态切换。
3. region 写入和查询。
4. 权限和登录保护。

API 按需验证：

1. 成功。
2. 失败。
3. 空数据。
4. 非法参数。
5. 返回结构兼容。
6. 敏感信息不泄露。

`advanced_analysis` 相关修改继续执行：

```bash
php80 tools/advanced_analysis_boundary_backtest.php
```

并记录澳门 / 香港最小样本、临界样本和大样本的结果、耗时和内存。

无法执行某项验证时：

1. 明确写“未验证”。
2. 说明环境限制。
3. 给出可复测步骤。
4. 不得写“全部通过”。

## 二十、部署与业务完整性

制作部署包、在线安装包或复制到其他站点时，必须先确定：

1. 空站安装包。
2. 业务完整安装包。

用户要求保留现有业务时，必须制作“业务完整安装包”，不能只打包程序骨架和空表。

业务完整安装包必须包含或安全重建：

1. 当前完整业务代码。
2. 所有必要模板、CSS、JavaScript 和系统静态素材。
3. 数据库 schema 和 migration。
4. 运行必需的业务基线数据。
5. 澳门 / 香港区域配置。
6. 页面、菜单、组件、分类和业务字典。
7. 功能开关。
8. 后台菜单、角色和必要权限基础配置。
9. 非敏感 API 数据源定义。
10. 定时任务定义。
11. 安装、升级、健康检查和回滚文档。

不得打包：

1. 用户密码和私人资料。
2. Session。
3. 生产日志和流量统计。
4. token、密钥和私钥。
5. 生产缓存。
6. 私人 uploads。
7. 生产数据库备份。

系统业务素材和用户私人 uploads 必须分开识别，不能机械排除整个资源目录。

构建前必须检查：

1. 未提交和未跟踪业务文件是否被遗漏。
2. `.deployignore` 是否误排除业务目录。
3. 构建脚本是否只打包 Git 已提交文件。
4. 业务基线数据是否完整。
5. API、定时任务和外部服务是否有安装配置。

安装验证必须对比源站与测试站：

1. 业务文件和哈希。
2. 数据库表结构。
3. 业务基线表行数。
4. 页面、菜单、组件和分类数量。
5. 澳门配置和香港配置。
6. 后台菜单和角色。
7. 系统业务素材。
8. 关键 API 返回结构。
9. 定时任务定义。
10. 登录、权限、数据隔离和开奖逻辑。

任何关键项不一致，不得说“业务完整”或“可上线”。

## 二十一、上线收敛与发布后观察

进入上线收敛后：

1. 停止新增非必要功能。
2. 清零 P0。
3. 处理影响上线的 P1。
4. P2 登记到后续，不在上线前顺手扩大。
5. 确认备份、部署、回滚和发布后检查清单。
6. 无真实验证不得说可上线。

发布后重点观察：

1. 澳门和香港关键页面。
2. 后台登录和业务入口。
3. API。
4. 404、500、PHP Fatal 和日志增长。
5. 数据库连接和慢查询。
6. CPU、内存、磁盘和 inode。
7. 数据串区。
8. 登录权限。
9. 开奖状态。

无法持续后台观察时，只提供检查清单，不得声称持续监控。

## 二十二、完成标准

任务只有同时满足以下条件才可以标记完成：

1. 用户要求的业务目标已实现。
2. 所有直接关联链路已补齐。
3. 澳门和香港按业务需要同步完成。
4. 共用后台按业务需要完成。
5. 数据读取、写入和区域隔离正确。
6. 空状态、错误状态和权限状态完整。
7. PC 和移动端可用。
8. 无新增 404、500、PHP Fatal 或明显 JavaScript 错误。
9. 未破坏登录、权限、接口契约和开奖独立。
10. 实际验证结果已记录。
11. 无法验证项已明确说明。
12. Git diff 与当前任务一致。

未达到时只能写：

1. 已完成部分。
2. 未完成项。
3. 阻断原因。
4. 下一步需要执行的唯一动作。

## 二十三、输出格式

回复应短、准、以结果为主，不得用长篇规则解释代替开发。

使用：

【本次业务】
当前完成的业务目标。

【实际完成】
列出关键交付结果。

【修改文件】
列出实际修改文件。

【澳门 / 香港 / 后台】
说明三者实际完成和影响。

【数据库 / 接口 / 登录 / 权限 / 开奖】
只说明实际变化或“未修改”。

【验证结果】
列出实际执行和结果。

【未验证或阻断】
没有则写“无”。

【Git状态】
未提交 / 真实提交号。

【本次任务状态】
完成 / 部分完成 / 阻断。

不必重复整份项目规则，不给无关下一步建议。
