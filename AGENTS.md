# 一、规则优先级

执行优先级：

1. 安全红线、数据安全和法律合规要求
2. 用户当前明确任务
3. 本仓库 AGENTS.md
4. 常驻自定义指令
5. Codex 默认行为

用户当前任务可以限定本次目标和范围，但不得覆盖安全红线。

如果 AGENTS.md 与常驻自定义指令冲突：

1. 安全红线始终优先。
2. 具体项目执行规则以本 AGENTS.md 为准。
3. 无法判断时停止高风险修改，说明冲突点，不得自行选择危险方案。

以下红线不得被覆盖：

1. 数据安全。
2. 澳门 / 香港数据隔离。
3. 登录与权限保护。
4. 开奖独立保护。
5. 禁止危险命令。
6. 禁止下注、赔率、彩票销售、购彩、提现、返奖、资金结算、赌博交易、赌博资金流转和非法引流。

如果“模块闭环”和“最小修改”冲突：

优先完成当前任务必要闭环，但不得跨无关业务扩大修改。

如果“普通模块边界”和“低风险批量修复”冲突：

1. 普通功能开发限定在同一个业务模块。
2. 低风险批量修复可以跨多个受影响文件或模块，但必须满足同一根因、同一风险等级、同一修复方式和同一验证标准。
3. 低风险批量修复不得涉及数据库结构、接口契约、登录权限、开奖核心或澳门 / 香港区域数据语义。

如果“继续推进”和“信息不足”冲突：

能从现有代码判断的，按现有代码推进。
不能从现有代码判断的，不得自由设计，必须说明缺失信息。

如果“UI 修复”和“稳定区域保护”冲突：

只允许修复当前问题，不允许重做稳定区域结构。

# 五、任务类型

一、只读排查

触发关键词：

排查、分析、检查、为什么、找原因、看日志、不要修改。

规则：

1. 只允许查看、搜索、分析。
2. 禁止修改任何文件。
3. 输出原因、证据、涉及文件、建议方案。


二、低风险修复

触发关键词：

修复、BUG、报错、异常、失效、不显示、错位、溢出、卡顿。

规则：

1. 只修改问题直接相关文件。
2. 不扩大范围。
3. 不顺手重构。
4. 不顺手优化无关模块。
5. 修复后必须验证对应页面或功能。


三、模块开发

触发关键词：

开发、新增、完成、补全、继续做、完善业务。

规则：

1. 按一个完整模块闭环推进。
2. 可以一次修改多个相关文件。
3. 相关文件必须属于同一个业务模块。
4. 不做碎片式修改。
5. 不跨无关模块。


四、UI 修复

触发条件：

用户发截图、指出视觉问题、说布局错乱、比例不协调、手机端异常。

规则：

1. 必须先定位对应代码。
2. 只修指定区域及必要相关区域。
3. 不得凭截图扩展到无关模块。
4. 不得重做设计。
5. 不得改变整体风格。


五、低风险批量修复

触发条件：

完成审计后，存在一组根因相同、风险边界清楚、可以统一验证的低风险问题。

规则：

1. 按问题组一次处理完整。
2. 不得混入数据库、接口契约、登录权限、开奖或区域数据等高风险变更。
3. 详细流程按“二十六、低风险批量修复流程”执行。


六、批量回归验证

触发条件：

完成低风险批量修复、跨多文件修改、上线收敛前复测。

规则：

1. 按影响矩阵批量验证，不重复执行无关测试。
2. 任一关键回归失败时停止进入下一阶段。
3. 详细流程按“二十七、批量回归验证流程”执行。


七、上线收敛模式

触发条件：

进入上线前冻结期，只允许处理上线阻断项和发布准备事项。

规则：

1. 停止新增非必要功能。
2. 优先清零 P0，上线前处理必要 P1。
3. 详细流程按“二十八、上线收敛模式”执行。


八、发布后观察期

触发条件：

生产发布完成，需要观察稳定性、日志、性能和业务链路。

规则：

1. 只观察、记录和处理明确生产异常。
2. 不得在生产环境中顺手重构或扩大修改。
3. 详细流程按“二十九、发布后观察期”执行。


# 六、禁止碎片式开发

禁止以下方式：

1. 只改一个按钮就结束。
2. 只改一行 CSS 就结束。
3. 只修一个角落不看当前模块整体。
4. 只做澳门不看香港。
5. 只做香港不看澳门。
6. 只做前台不看后台。
7. 只写页面不接数据。
8. 只接数据不处理空状态。
9. 只做 PC 不看手机端。
10. 做一步停下来问一步。

正确方式：

1. 一个页面、一个功能、一个后台模块、一个 UI 区块，必须形成当前任务范围内的完整闭环。
2. 当前需求直接相关的前台、后台、API、数据处理、样式、空状态、错误状态、移动端适配，应一次处理完整。
3. 能从现有代码和同类页面判断的，直接按现有结构推进。
4. 信息不足时，不允许自由设计，只能基于现有代码和同类页面做最小一致性补全。
5. 只有高风险事项才停下来确认。


# 七、修改范围规则

普通功能开发允许一次修改多个相关文件，但必须属于同一个业务模块。

低风险批量修复可以涉及多个受影响文件或模块，但必须同时满足：

1. 同一根因。
2. 同一风险等级。
3. 同一修复方式。
4. 同一验证标准。
5. 不改变业务语义。
6. 不涉及数据库结构、接口契约、登录权限、开奖核心或澳门 / 香港区域数据语义。

禁止：

1. 跨无关模块顺手修改。
2. 顺手重构。
3. 顺手优化。
4. 批量格式化全项目。
5. 批量替换未知范围代码。
6. 删除未确认文件。
7. 清空日志。
8. 清空缓存。
9. 删除上传目录。
10. 改动未确认的稳定区域。

普通功能开发预计修改超过 3 个业务文件时，必须先说明原因和必要性。

低风险批量修复不以文件数量强制拆碎，但必须先列出问题组、涉及文件、统一修复方式、验证标准和回滚方式。


# 八、澳门 / 香港双前台规则

必须始终同时考虑澳门和香港。

规则：

1. 澳门有的功能，必须检查香港是否需要对应。
2. 香港有的功能，必须检查澳门是否需要对应。
3. 共用后台必须同时服务澳门和香港。
4. 数据必须使用项目现有 region / site / system 等区域字段区分。
5. 不得让澳门数据覆盖香港数据。
6. 不得让香港数据覆盖澳门数据。
7. 不得只修澳门漏掉香港。
8. 不得只修香港漏掉澳门。
9. 不得无理由制造两套不同代码。
10. 前台可以有地区文案差异，但结构和功能逻辑必须保持一致。

允许不同：

1. 系统名称
2. 品牌标识
3. 地区文案
4. 开奖时间
5. 开奖期号
6. 开奖结果
7. 开奖数据来源
8. 开奖历史记录


# 九、开奖独立规则

开奖逻辑是高风险区域。

未经用户明确确认，不得修改开奖核心逻辑。

禁止：

1. 合并澳门 / 香港开奖结果。
2. 共用澳门 / 香港开奖时间。
3. 共用澳门 / 香港开奖期号。
4. 因后台共用导致开奖数据串区。
5. 不涉及开奖时误改开奖逻辑。

凡是涉及开奖，必须明确说明：

1. 是否影响澳门开奖。
2. 是否影响香港开奖。
3. 是否影响开奖时间。
4. 是否影响开奖期号。
5. 是否影响开奖结果。
6. 是否影响开奖数据来源。
7. 是否影响历史记录。

# 十二、UI / 布局 / 视觉规则

UI 修改必须以以下来源为准：

1. 现有代码结构
2. 用户截图明确指出的问题
3. 同类页面已有样式
4. 项目现有 CSS 规则

禁止：

1. 凭感觉重新设计。
2. 改变页面整体结构。
3. 改变模块顺序。
4. 改变信息层级。
5. 改卡片结构。
6. 改字体体系。
7. 改整体风格。
8. 大量使用 !important。
9. 随意修改全局 CSS。
10. 澳门一套 UI，香港一套 UI。
11. 为了美观破坏功能。
12. 为了塞内容把字体改得过小。
13. 用固定高度硬压动态内容。
14. 图片拉伸变形。

允许：

1. 修错位。
2. 修溢出。
3. 修间距。
4. 修对齐。
5. 修图片变形。
6. 修手机端显示异常。
7. 在同类页面已有样式基础上补齐缺失样式。

UI 修复不能只改一个点，必须检查当前模块整体协调：

1. 顶部是否正常。
2. 内容区是否正常。
3. 卡片比例是否正常。
4. 字体层级是否正常。
5. 按钮大小是否正常。
6. 图片是否变形。
7. 列表是否错位。
8. 底部导航是否遮挡内容。
9. 手机端是否横向溢出。
10. 澳门 / 香港对应页面是否一致。


# 十三、移动端标准

移动端优先。

必须满足：

1. 页面不得横向滚动。
2. 主体内容不得贴边。
3. 底部导航不得遮挡内容。
4. 点击区域不得过小。
5. 文字不得挤成一团。
6. 卡片不得被压扁。
7. 长标题必须换行或省略，不能撑爆布局。
8. 图片不得拉伸变形。

常用尺寸参考：

1. 页面左右边距：10px - 14px。
2. 卡片内边距：8px - 12px。
3. 模块上下间距：8px - 16px。
4. 正文字号：13px - 15px。
5. 辅助文字：12px - 13px。
6. 标题字号：16px - 20px。
7. 重要数字：18px - 26px。
8. 按钮高度：36px - 44px。
9. 底部导航点击区不低于 44px。


# 十四、CSS / HTML 治理规则

CSS 修改必须遵守优先级：

1. 组件 CSS
2. 模块 CSS
3. 页面 CSS
4. 全局 CSS

禁止优先修改全局 CSS，除非问题确实来自全局样式。

禁止：

1. 大量新增 !important。
2. 重复创建功能相同的 class。
3. 多个文件反复覆盖同一个选择器。
4. 用临时 CSS 掩盖 HTML 结构问题。
5. 全站搜索替换未知影响的 class。
6. 为一个页面问题改全局布局。
7. 批量重排 CSS 文件。
8. 批量格式化 CSS 文件。

后台字体权重必须复用公共变量：

1. 后台表头正文统一使用 `--admin-head-font-weight`，当前标准值为 `600`。
2. 后台表单控件正文统一使用 `--admin-control-font-weight`，当前标准值为 `500`。
3. 后续新增或修改后台组件时，禁止为上述两类正文新增局部 `font-weight` 数值硬编码。
4. 按钮、徽章、状态和业务强调文本不属于上述两类正文，必须按既有组件语义维护，不得借公共规则顺带修改。
5. 确需例外时，必须先说明原因、选择器范围和兼容影响，不得用更高优先级覆盖公共变量。

HTML 修改必须遵守：

1. 不新增无意义 div 嵌套。
2. 不复制大量重复结构。
3. 不改变原有模块顺序。
4. 不改变信息层级。
5. 不破坏澳门 / 香港结构一致性。
6. 不用 CSS 掩盖 HTML 结构错误。

发现 CSS 或 HTML 重复时，不能直接大清理，必须先说明：

重复位置：
影响范围：
建议保留方案：
删除风险：
是否本次必须处理：

# 十六、后台开发规则

后台是共用后台，必须服务澳门和香港。

后台模块必须考虑：

1. 列表显示。
2. 分页。

# 二十六、低风险批量修复流程

低风险批量修复用于处理一组根因相同、影响边界明确、可统一验证的问题，避免逐个文件碎片式处理。

## 进入条件

同时满足以下条件才允许进入：

1. 已完成只读排查或审计。
2. 已形成明确的问题清单。
3. 问题根因相同或修复方式一致。
4. 涉及范围、引用关系和回滚方式已经确认。
5. 不涉及数据库结构、接口契约、登录权限、开奖核心和澳门 / 香港数据语义。
6. 当前工作区已有修改已识别并受到保护。

## 允许批量处理

可以按同一问题组一次处理：

1. 明确的 404 静态资源引用。
2. 不影响业务语义的同类 PHP Notice 或 Warning。
3. 重复加载的 CSS、JavaScript 或静态资源。
4. 同一根因造成的重复事件绑定或重复请求。
5. 明确无效的调试输出、临时日志和开发残留。
6. 明确被覆盖且无实际生效作用的重复 CSS 规则。
7. 明确无引用、可恢复、非用户数据的临时文件或编辑器残留。
8. 同一组件内可证明等价的低风险重复代码。

## 禁止纳入批量修复

以下内容必须退出批量模式，按对应高风险规则处理：

1. 数据库结构、字段含义和生产数据。
2. API 地址、请求方式、参数和返回结构。
3. 登录、Session、权限和后台访问控制。
4. 澳门 / 香港 region、site、system 隔离逻辑。
5. 开奖时间、期号、结果、来源和历史记录。
6. uploads、用户数据、证书、密钥和当前生产配置。
7. 无法证明无引用的文件、函数、模板和资源。
8. 需要重构架构才能处理的问题。

## 执行流程

1. 为当前批次定义唯一“问题组名称”。
2. 列出涉及文件、根因、统一修复方式和回滚点。
3. 执行 `git status --short`，保护已有修改。
4. 同一问题组一次处理完整，不混入其他问题。
5. 不进行无关格式化，不顺手清理其他代码。
6. 完成后检查：
   - `git diff --stat`
   - `git diff --name-status`
   - 当前问题组的实际 diff
7. 立即执行受影响模块验证。
8. 当前问题组验证通过后，才进入下一问题组。
9. 未经用户明确要求，不自动 commit、push、部署或重启服务。

涉及文件较多时，不以文件数量强制拆碎；但必须保持同一根因、同一风险级别和同一验证方法。

## 失败处理

出现以下任一情况时，立即停止当前批次：

1. 发现影响范围超出原问题组。
2. 发现需要修改高风险区域。
3. 产生大面积非预期 diff。
4. 澳门 / 香港任一对应功能出现差异。
5. API、登录、权限、开奖或数据隔离出现变化。
6. 无法证明修复前后业务行为等价。

停止后必须报告已修改文件、失败原因、风险和安全回退建议，不得擅自执行危险回滚命令。

## 完成标准

当前问题组只有在以下条件全部满足时才算完成：

1. 同类问题已在约定范围内处理完整。
2. 无新增 404、500、PHP Fatal 或 JavaScript 错误。
3. 业务流程和接口契约未改变。
4. 澳门 / 香港数据隔离未改变。
5. 登录、权限和开奖逻辑未改变。
6. 相关回归验证通过。
7. Git diff 清晰、可审查、可回滚。


# 二十七、批量回归验证流程

批量回归验证用于低风险批量修复、多文件修改和上线前复测。它负责验证变更影响矩阵，不替代“二十五、验证规则”，而是在其基础上统一组织测试。

## 触发条件

出现以下任一情况必须执行：

1. 完成一个低风险批量修复问题组。
2. 同一任务修改多个相关文件。
3. 同时涉及 PHP、HTML、CSS、JavaScript 或 API 中的两个及以上层级。
4. 修改澳门 / 香港共用模板、共用组件或后台公共逻辑。
5. 进入上线收敛模式。
6. 修复 P0 或 P1 问题后。

## 回归矩阵

先根据实际变更建立矩阵：

1. 变更文件。
2. 直接受影响模块。
3. 间接受影响模块。
4. 澳门对应入口。
5. 香港对应入口。
6. 共用后台入口。
7. API 或数据读取链路。
8. PC 和移动端视口。
9. 登录、权限、开奖和数据隔离是否需要保护性验证。

只执行与实际影响有关的深度测试，但以下冒烟项不得省略：

1. 澳门对应页面可访问。
2. 香港对应页面可访问。
3. 共用后台相关入口可访问。
4. 无新增 404、500、PHP Fatal。
5. 无新增 JavaScript 控制台错误。
6. 澳门 / 香港数据不串区。
7. 登录、权限和开奖未被意外影响。
8. `git status --short` 与实际修改一致。

## 执行顺序

1. 静态检查和语法检查。
2. 直接受影响功能测试。
3. 澳门 / 香港对应功能对照测试。
4. 共用后台相关链路测试。
5. API 成功、失败、空数据和非法参数场景。
6. PC 与移动端响应式测试。
7. 日志、404、500、Fatal 和重复请求检查。
8. Git diff 与未跟踪文件检查。

不得使用不可回滚的生产写入来完成回归测试。需要写入测试但没有安全测试条件时，必须标记“未执行写入测试”。

## 失败处理

任一关键回归失败时：

1. 当前任务不得标记为完成。
2. 不得进入上线收敛或发布阶段。
3. 记录失败入口、复现步骤、错误日志和相关文件。
4. 判断是本次回归、既有问题还是环境限制。
5. 只修复本次变更引入的问题；既有无关问题另行登记。
6. 修复后重新执行失败项和必要冒烟项。

## 证据要求

回归结果必须注明：

1. 实际执行的命令。
2. 实际访问的页面或接口。
3. 实际测试的视口。
4. 实际检查的日志。
5. 通过、失败、未验证的项目。
6. 未验证原因。

不得仅写“测试通过”而没有可核对的测试范围。


# 二十八、上线收敛模式

上线收敛模式是发布前的功能冻结与风险清零阶段。进入本模式后，不再继续普通功能扩展。

## 进入条件

1. 当前计划上线的业务范围已经明确。
2. 主要功能开发已完成。
3. 已有问题按 P0、P1、P2 分级。
4. 已确认当前发布分支、stable 基线和 Git 状态。
5. 已具备数据库、项目文件和 uploads 的备份或明确备份方案。

## 收敛原则

1. 冻结新增非必要功能。
2. P0 必须清零。
3. 影响上线的 P1 必须修复或形成经用户确认的接受结论。
4. P2 只登记，不在上线前顺手扩大处理。
5. 不进行架构重构、视觉重做和技术栈迁移。
6. 不改变数据库和 API 契约，除非用户单独批准。
7. 所有修复必须可回滚并通过批量回归验证。

## 执行流程

1. 确认当前发布范围和不发布范围。
2. 执行项目识别、Git 检查和 stable 基线核对。
3. 汇总未完成项、P0、P1、P2 和未验证项。
4. 按优先级处理：
   - 500、白屏、PHP Fatal
   - 登录与权限异常
   - 澳门 / 香港数据串区
   - 开奖异常
   - 关键 API 和后台操作异常
   - 严重移动端或导航问题
   - 安全和部署阻断项
5. 每个修复问题组执行批量回归验证。
6. 检查敏感文件、日志、缓存、临时文件和发布排除规则。
7. 确认备份、部署步骤、配置差异和回滚方案。
8. 输出最终上线判断：
   - 可上线
   - 有条件上线
   - 禁止上线

## 上线门槛

满足以下条件才允许建议“可上线”：

1. 无未处理 P0。
2. 无已知数据损坏或串区风险。
3. 登录、权限和开奖关键链路验证正常。
4. 关键 API 和后台业务链路正常。
5. 无持续 500、PHP Fatal 或日志刷屏。
6. 主要 PC 和移动端入口无阻断问题。
7. 数据库、项目文件、uploads 的备份和恢复路径明确。
8. 发布与回滚步骤明确。
9. Git diff、提交范围和未跟踪文件已审查。
10. 所有无法验证项已明确，不得隐瞒。

## 发布操作边界

未经用户明确授权，不自动执行：

1. commit。
2. push。
3. tag。
4. 部署。
5. 数据库迁移。
6. 服务重启。
7. 缓存清理。
8. 生产回滚。


# 二十九、发布后观察期

发布后观察期用于确认新版本在生产环境中的真实稳定性，不等同于继续开发。

## 默认观察窗口

根据项目现有监控能力执行以下观察点：

1. 发布后立即检查。
2. 发布后 30 分钟内。
3. 发布后 2 小时内。
4. 发布后 24 小时内。
5. 重要版本继续观察至 72 小时。

无法持续在线观察时，必须为用户提供每个观察点的检查清单和命令，不得声称后台持续监控。

## 观察内容

1. 澳门首页、香港首页和关键前台入口。
2. 后台登录和关键管理入口。
3. 澳门 / 香港开奖时间、期号、结果和来源。
4. 澳门 / 香港数据隔离。
5. API 成功率、错误返回和响应异常。
6. HTTP 404、500。
7. PHP Fatal、Warning 和异常日志增长。
8. JavaScript 错误和重复请求。
9. 数据库连接、慢查询和写入异常。
10. CPU、内存、磁盘、inode 和进程状态。
11. 登录失败、权限异常和安全事件。
12. 静态资源缺失、缓存版本和页面显示异常。
13. 用户关键路径和后台关键业务流程。

## 异常分级

立即建议停止扩展并进入故障处理的情况：

1. 持续 500、白屏或 PHP Fatal。
2. 登录或后台权限异常。
3. 澳门 / 香港数据串区。
4. 开奖时间、期号、结果或来源异常。
5. 关键 API 大面积失败。
6. 数据写入错误或可能造成数据损坏。
7. 上传、安全或敏感信息泄露风险。
8. CPU、内存、磁盘或日志异常增长影响服务。

一般问题：

1. 非阻断 UI 差异。
2. 个别非核心静态资源问题。
3. 不影响业务的文案或轻微兼容问题。

一般问题应登记为后续任务，不得在生产环境中顺手重构。

## 处理规则

1. 先保留证据：时间、入口、错误、日志、请求和版本。
2. 判断问题是否由本次发布引入。
3. 只处理明确的生产阻断问题。
4. 任何修复仍须遵守 Git、权限、数据库、开奖和回滚规则。
5. 满足回滚条件时，立即停止继续修改并提出回滚建议。
6. 未经用户授权，不自动执行 git reset、强制回滚、数据库恢复或服务重启。

## 观察期结束条件

满足以下条件才可以结束观察期：

1. 关键业务入口持续正常。
2. 无新增 P0。
3. 无持续 500、PHP Fatal 或日志刷屏。
4. 澳门 / 香港数据隔离正常。
5. 登录、权限、开奖和关键 API 正常。
6. 服务器资源无异常趋势。
7. 已记录全部遗留问题和后续处理优先级。

结束时必须输出：

【发布版本】
【观察时间范围】
【检查入口】
【日志与性能状态】
【澳门 / 香港状态】
【登录 / 权限 / 开奖状态】
【异常与处理】
【是否触发回滚条件】
【遗留问题】
【最终稳定性结论】

# 三十一、二次开发工作方式

本项目是半成品二次开发，不是从零开发。

默认工作方式：

1. 先收敛现有混乱。
2. 再补齐缺失功能。
3. 再统一澳门 / 香港。
4. 再完善后台闭环。
5. 最后才做优化和美化。

禁止：

1. 推翻重做。
2. 新旧结构并存。
3. 每次新增一套写法。
4. 为了快制造临时代码。
5. 为了美观破坏业务。
6. 为了性能提前重构。

正确方式：

1. 找已有同类模块。
2. 复用已有结构。
3. 补齐当前模块缺口。
4. 保持澳门 / 香港一致。
5. 保持后台共用。
6. 保持数据隔离。
7. 完成后验证。


# 三十二、核心纪律

不重构，只演进。
不乱猜，只看代码。
不碎改，只做任务包或问题组闭环。
不自由设计，只基于现有结构修复。
不只做澳门漏香港。
不只做香港漏澳门。
不只做前台漏后台。
未经明确确认，不修改登录和权限；经确认后也只做必要的最小修改。
未经明确确认，不修改开奖核心逻辑；经确认后仍保持澳门 / 香港开奖独立。
不编造结果。
不假装验证。
不扩大无关范围。
不破坏 stable。
